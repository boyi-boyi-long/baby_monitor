## 1. 聲音線 — 哭聲偵測（警報主幹）

### 1.1 核心技術

聲音線用的**是深度學習**：YAMNet（Google 預訓練的音訊分類模型，521 種聲音類別，含 `Baby cry, infant cry`／`Crying, sobbing`／`Whimper`）。

```
INMP441 麥克風 → I2S 16kHz 取樣 → ESP32 → UDP → 筆電環形緩衝區（存最近 1 秒）
→ 每 0.5 秒：取最近 1 秒波形 → 丟進 YAMNet 推論
→ 521 類分數中，取監看類別的最高分 = cry_score
→ 滑動視窗（最近 6 次判斷）多數決：6 次中 ≥4 次超過門檻 → 觸發警報
→ 觸發後清空視窗 + 冷卻 120 秒（防止同一次哭聲連環轟炸）
```

YAMNet 是**已經訓練好、直接套用**的現成模型（凍結權重，只做推論不做訓練），不需要自己準備哭聲錄音資料集。

### 1.2 為什麼放棄前一版方案

聲音線其實經歷過**兩次**方案更換，都是實際試過、失效或有明確缺陷後才換掉的：

#### (a) 麥克風硬體：MAX9814（類比）→ INMP441（I2S 數位）

| | MAX9814（捨棄） | INMP441（現用） |
|---|---|---|
| 訊號型態 | 類比電壓輸出，需經 ESP32 內建 ADC 轉數位 | I2S 數位匯流排，直接輸出數位樣本 |
| 已知問題 | 1. 內建 **AGC（自動增益控制）**：晶片自己動態調整增益，導致同樣音量在不同時刻被放大的倍率不一致，波形振幅不穩定，直接影響「這段聲音有多大聲」的判斷基準<br>2. ESP32 **內建 ADC 本身噪聲大、非線性**，類比訊號經過它會失真<br>3. 取樣率是靠軟體 `analogRead()` 計時，**沒有硬體時脈保證**，實際取樣間隔會抖動（jitter），影響音訊在時間軸上的準確性 |
| 改善點 | I2S 是硬體時脈驅動的數位匯流排，取樣率由硬體保證精確、不抖動；訊號在麥克風內部就轉成數位，不經過 ESP32 的類比前端，繞開 ADC 噪聲；**沒有 AGC**，音量大小是麥克風忠實反映的原始強度，YAMNet 拿到的波形更接近「真實聲音」 |

簡言之：MAX9814 的類比路徑會在「聲音變成數字」的過程中引入好幀不可控的失真與抖動，而哭聲偵測的準確度高度依賴輸入波形的品質，所以換成訊號鏈更乾淨的數位麥克風。

#### (b) 推論位置與模型：ESP32 + Edge Impulse 自訓模型（曾嘗試）→ 筆電端 YAMNet（現用）

早期方案是用 **Edge Impulse** 幫忙訓練一個小型哭聲分類模型，直接燒錄進 ESP32，讓感測端自己判斷「這是不是哭聲」，不用把音訊傳到筆電。

**實際部署後偵測失效**，原因是典型的 **domain shift（訓練/部署條件不一致）**：

- 訓練資料的錄音環境（麥克風型號、房間迴響、收音距離、環境噪音）跟實際部署時不完全一致
- 微控制器等級的小模型本身參數量少、泛化能力弱，對這種環境差異特別敏感
- 結果：模型在訓練資料上表現正常，換到真實嬰兒房環境就大量漏判/誤判

**改用 YAMNet 的原因**：它是 Google 用海量、多樣化真實世界聲音資料訓練出的大型通用音訊模型，對麥克風型號、環境差異的抗性遠高於自訓小模型；缺點是模型較大，跑不動在 ESP32 上，因此連帶把「推論該放在哪裡」這個架構決策一起定案：

> **感測端零智慧、推論全在筆電**——ESP32 只負責採集與搬運原始資料，所有判斷都在運算資源充足的筆電端進行。

（這兩個教訓也回頭影響了「明確不做」清單：不在 ESP32 上跑任何 ML 模型，見 `doc/PRD.md` 第 7 節 D1。）

## 2. 影像線 — 活動量偵測（「可能醒了」預警）

### 2.1 核心技術

影像線**沒有用深度學習**，用的是傳統影像處理演算法——**幀間差異法（frame differencing）**，純數學規則、無模型無訓練：

```
StreamReader 常駐讀 ESP32-CAM 的 MJPEG 串流，永遠只保留「最新一張」畫面
→ 每秒一次：
    gray_now  = 這一幀轉灰階
    diff      = |gray_now - gray_prev|（逐像素亮度差絕對值）
    motion_ratio = (diff > 25 的像素數) / 總像素數   ← 這一秒「變動比例」
    active    = motion_ratio > 門檻                  ← 這一秒算不算「有在動」
→ 存進最近 3 分鐘的視窗（180 個 True/False）
→ 3 分鐘視窗內「有在動」比例超標，且最近 30 秒仍在動 → 📱「可能醒了」預警
→ 若曾經升高但最近 30 秒已恢復平靜 → 視為短暫扭動，不通知（對應 US-3）
```

判斷「有沒有動」完全不需要知道「這是寶寶的手還是被子」，只看畫面亮度變化幅度，計算量極輕（純數值運算），不需要 GPU、不需要模型檔案。

### 2.2 為什麼沒有採用姿勢辨識（MediaPipe），而是用幀差法

影像線在設計階段就評估過用 **MediaPipe** 之類的姿勢辨識（pose landmark）模型，去判斷寶寶的睡姿/清醒動作，但最終**沒有採用**，原因：

| 考量點 | MediaPipe 姿勢分類（未採用） | 幀差活動量（現用） |
|---|---|---|
| 遮擋問題 | 嬰兒常被包在被子、衣物、床圍裡，身體關鍵點（手、腳、軀幹）大範圍被遮擋，姿勢模型抓不到 landmark 或抓錯，判斷不穩定 | 不需要辨識任何身體部位，被子動、手動、翻身動——只要畫面亮度有變化就算數，對遮擋完全不敏感 |
| 嬰兒體型 | 姿勢模型多半以成人骨架比例訓練，嬰兒的體型比例差異大，容易誤判 | 不涉及體型/比例假設 |
| 運算成本 | 需要跑模型推論，對筆電資源要求較高 | 純數值運算，資源需求極低 |
| 判斷粒度 | 想知道「姿勢是什麼」，但實際上產品只需要「有沒有可能醒了」這個粗粒度資訊 | 剛好對應實際需求：夠用、不過度設計 |

判斷邏輯是：**產品真正要回答的問題是「寶寶是不是在持續活動、可能醒了」，不是「寶寶現在是什麼姿勢」**。姿勢辨識能給的資訊比需求多，但穩定性換來的代價（遮擋失效）不划算；幀差法資訊量剛好打平需求，反而更穩健。最終判斷是否要通知家長，交給截圖讓人眼判斷，不需要機器精準分類姿勢。

（對應 `doc/SPEC.md` 第 8 節 D5：第一版不做姿勢分類，由「活動量預警＋截圖」覆蓋，避免 MediaPipe 在嬰兒體型/遮擋下的不穩定。）

## 3. 兩條線總覽對照

| | 聲音線（哭聲偵測） | 影像線（活動量偵測） |
|---|---|---|
| 是否為深度學習 | 是（YAMNet） | 否（傳統幀差演算法） |
| 曾嘗試過、後來放棄的方案 | ① MAX9814 類比麥克風 → INMP441 數位麥克風<br>② ESP32+Edge Impulse 自訓模型 → 筆電 YAMNet | MediaPipe 姿勢辨識（評估後未採用，非「用過又換掉」） |
| 放棄/未採用的核心原因 | ① AGC／ADC 噪聲／取樣率抖動，訊號品質不穩<br>② domain shift 導致部署後偵測失效 | 嬰兒遮擋情境下 landmark 偵測不穩定，且超出實際需求粒度 |
| 現行方案的關鍵優勢 | 硬體時脈保證取樣精確、無 AGC 干擾；大型通用模型對環境差異抗性強 | 對遮擋、體型不敏感；運算成本趨近於零；粒度剛好對應「有沒有醒」的需求 |
| 角色定位 | 警報主幹（哭聲是最可靠的求救訊號） | 輔助預警＋提供截圖，最終判斷交給看截圖的家長 |

**共通設計理念**：兩條線都遵守「感測端零智慧、推論/運算全在筆電」——差別只在於聲音線的「運算」是深度學習推論，影像線的「運算」是傳統演算法，兩者都不在 ESP32 上執行，ESP32 永遠只負責採集與搬運原始資料。
